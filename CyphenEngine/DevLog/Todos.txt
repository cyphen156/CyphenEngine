## 해야 할 일

1. Time Class
	- PlatformTime HAL 분리는 완료된 상태입니다.
	- Core Time은 엔진 루프 기준 DeltaTime / FPS / 누적 시간 계산만 담당합니다.
	- 엔진 루프가 별도 스레드로 분리된 구조를 기준으로 초기화 / 업데이트 위치를 재검토합니다.
	- Run 루프에 Time::Update 또는 임시 yield 처리를 추가합니다.
	- 멀티스레드 분기 가능성을 고려합니다.

2. File Class
	- 기본적인 단일 파일 동기 I/O는 정리된 상태입니다.
	- File은 bool 반환과 out 파라미터를 사용하는 공개 저수준 API입니다.
	- File은 TaskSystem / ResourceManager / FileSystem을 알지 않습니다.
	- PlatformFile은 HAL/Private 내부 구현으로 분리된 상태입니다.
	- Windows API 문자열 경계는 WindowsString이 담당합니다.
	- Linux API 문자열 경계는 LinuxString / LinuxPath가 담당합니다.
	- CString과 raw byte 사이의 텍스트 인코딩 변환은 TextCodec이 담당합니다.
	- 파일 중간에 데이터 추가하는 기능은 추후 확장합니다.
	- 파일 확장자 유형 기반 처리는 ResourceManager 또는 FileSystem 계층에서 검토합니다.
	- 비동기 파일 로딩은 File 클래스 자체가 아니라 ResourceManager / TaskSystem 계층에서 처리합니다.

3. Path Class
	- 순수 경로 문자열 유틸리티 구현은 완료된 상태입니다.
	- Normalize / Combine / GetDirectoryName / GetFileName / GetFileNameWithoutExtension / GetExtension을 제공합니다.
	- Path는 프로젝트 루트 / Content / Saved / Logs 같은 엔진 정책 경로를 보관하지 않습니다.
	- PlatformPath는 engine path를 현재 플랫폼 경로 규칙의 CString으로 직렬화하는 단방향 HAL 계약으로 정리된 상태입니다.
	- WindowsPath / LinuxPath는 PlatformPath와 플랫폼 문자열 변환을 묶는 native path facade입니다.
	- Windows 특수 디렉터리, 실행 파일 기준 경로, 현재 작업 디렉터리 등이 필요해지는 시점에 PlatformDirectory 또는 PlatformKnownPath 계층을 별도로 검토합니다.
	- 이후 Content / Project / Engine / Saved / Temp 경로 체계를 만들기 위한 상위 정책 계층을 검토합니다.

4. Logger Class
	- 메인 스레드 처리 또는 로그 메시지 큐 스레드 분기 처리를 고려합니다.
	- 플랫폼별 출력 처리를 분리합니다.
	- Windows는 OutputDebugStringA, Linux는 stderr 출력 경로를 기본 debug sink로 둡니다.
	- Fatal / Internal 계열 로그는 크래시 추적을 위해 즉시 출력 및 중단 처리 방향을 고려합니다.
	- 이후 Diagnostics 계층으로 LogRecord 생성과 큐 적재 책임을 분리할지 검토합니다.
	- 디버그용 윈도우를 만들 경우 해당 창에 텍스트 렌더링을 연결할 예정입니다.

5. ResourceManager / FileSystem
	- File이 제공하는 동기 I/O API를 TaskSystem / WorkerThread에 넘겨 비동기 로딩 구조를 구성합니다.
	- 파일 존재 여부, 삭제, 복사, 이동, 디렉터리 생성 / 삭제 같은 파일 시스템 관리 기능을 별도 계층으로 분리합니다.
	- AssetId / ResourceHandle / PakHandle / Offset 기반 접근 구조를 검토합니다.
	- 런타임 hot path에서는 문자열 path 기반 반복 접근을 피하도록 설계합니다.
	- Texture2D CPU decode와 GPU upload 경계를 ResourceCommand / RendererCommand 흐름 기준으로 유지합니다.

6. Renderer
	- 렌더러는 Renderer Module ABI를 통해 런타임 적재되는 선택 모듈로 구성합니다.
	- Renderer 모듈은 필수 엔진 구성요소가 아닙니다.
	- Server / headless 빌드에서는 Renderer 모듈을 비활성화할 수 있어야 합니다.

	- 구현체:
		- CyphenRendererDx11.dll
		- CyphenRendererVulkan.dll / .so
		- CyphenRendererOpenGLES는 보류 슬롯으로 유지합니다.
		- Metal은 현재 우선순위에서 제외합니다.

	- 완료된 항목:
		- Dx11 Renderer Backend DLL 분리
		- Renderer Module ABI 구성
		- RenderCommand / ResourceCommand 기반 Texture2D 업로드와 DrawTexturedQuad 실행
		- Windows Vulkan first-light
		- Vulkan 렌더러에서 Texture2D 두 장 교대 출력 검증

	- 남은 항목:
		- Linux CyphenRendererVulkan.so 빌드 구성
		- Linux Vulkan surface 생성 경로 정리
		- Vulkan 출력의 상하좌우 반전 원인 정리 및 수정 여부 판단
		- swapchain recreate / resize 대응
		- renderer backend별 command decode 중복 정리 여부 검토

	- 설계 검토:
		- Windows는 HWND 기반 windowed entry가 OS 앱 모델에 자연스럽게 포함됩니다.
		- Linux는 GUI가 기본 실행 조건이 아니므로 X11 / Wayland 의존성을 Platform/Linux 기본값으로 고정하지 않습니다.
		- X11 / Wayland는 windowed target 또는 renderer 출력 경로의 선택 의존성으로 관리합니다.
		- Linux renderer .so가 display / window / surface를 자체 소유할지, Launch가 windowed entry로 제공할지 추가 검토합니다.

7. Linux Porting
	- Linux PlatformFile / PlatformPath / PlatformTime / ModuleLoader 구현은 완료된 상태입니다.
	- LinuxJpegCodec은 libjpeg-turbo 기반 decode leaf로 구현된 상태입니다.
	- Linux 실행 파일은 WSL 환경에서 CoreIoTests PASS=69 FAIL=0까지 검증된 상태입니다.
	- 다음 목표는 Linux Renderer 모듈 .so 빌드 / 산출물 배치 / export 주입입니다.
	- X11 Launch는 Linux first-light 검증을 위한 windowed 경로로 봅니다.
	- headless / server 실행 경로와 windowed renderer 실행 경로를 분리할지 검토합니다.

8. Build Target / Runtime Config
	- BuildInfo를 CyphenEngine::Run 내부 BUILD_TARGET 분기에 실제 연결합니다.
	- Renderer 모듈 활성화 여부를 빌드 설정과 실행 설정 양쪽에서 다룰 수 있도록 정리합니다.
	- Server / Editor / Windowed 타겟 구분을 Build 계층에서 명확히 합니다.
	- RuntimeConfig는 렌더러 선택 / PIE / Standalone Play / Multiplay Mode 같은 런타임 옵션을 다룰 때 추가합니다.
	- Renderer 변경은 런처 재시작 또는 프로세스 재시작이 필요한 방향으로 둡니다.

9. Event / Window
	- WndProc → PlatformEventQueue → Engine Tick 소비 구조를 추가합니다.
	- Windows message loop와 Linux X11 event loop를 동일한 상위 이벤트 계약으로 올릴지 검토합니다.
	- NativeWindowInfo는 현재 windowed renderer 초기화를 위한 최소 전달 구조입니다.
	- Renderer 모듈 또는 별도 WindowSystem 계층이 platform window / surface를 직접 소유하는 구조가 확정되면 NativeWindowInfo 책임을 축소하거나 제거합니다.
